# 08 Glosario

## Aviso de ambigüedad de nombres (leer primero)

- **`Axiom.Spec/`** (mayúsculas, este repo): fuente de verdad documental del WORKSPACE de desarrollo de Axiom (specs 00-08, incrementos, bugs, decisiones).
- **`axiom.spec/`** (minúsculas): carpeta que el PRODUCTO Axiom espera/genera dentro de cualquier proyecto que lo adopte, con CONTENIDO DE SPEC (`templates/`, `target-axiom-skills/`, `target-axiom-agents/`, `technical-context/`, `product.manifest.yaml`), incluido potencialmente el propio repo `Axiom/` (dogfooding) — hoy ausente en ese checkout. Ver [context/references/03-riesgos-y-brechas-conocidas.md](../context/references/03-riesgos-y-brechas-conocidas.md). Desde `INC-20260703-config-folder-renames`, YA NO contiene `config/` (ver `axiom.config/` abajo) — es una carpeta hermana, no anidada.
- **`axiom.config/`** (minúsculas): carpeta de CONFIG generada/declarativa que el PRODUCTO Axiom espera/genera en la raíz de cualquier proyecto que lo adopte (`topology.yaml`, `profiles.yaml`, `providers.yaml`, `capabilities.yaml`, y ~15 YAML más — ver [03_Modelo_Operativo_y_Datos.md](03_Modelo_Operativo_y_Datos.md)). Renombrada desde `axiom.spec/config/` en `INC-20260703-config-folder-renames`: es config, no contenido de spec, y ya no vive anidada dentro de `axiom.spec/`.
- **`.axiom-state/`** (minúsculas): carpeta de ESTADO RUNTIME project-scoped que `axiom init` y comandos posteriores generan en la raíz del proyecto adoptante (`init.json`, `install-profile.json`, `members.yaml`, overlay local en `.axiom-state/local/`, checkpoints). Renombrada desde `.sdd/` en `INC-20260703-config-folder-renames`. No confundir con `axiom.spec/`/`axiom.config/`, que son declarativos/versionados, no estado runtime.
- Nunca usar estos términos como sinónimos en la documentación.

## Términos del producto

- **Builder tooling**: tooling usado para diseñar y construir el propio Axiom.
- **Product model**: representación interna de Axiom para workflows, capabilities y políticas.
- **Generated configuration**: archivos de entorno generados a partir del modelo de Axiom.
- **Capability**: función independiente de plataforma expuesta por Axiom, con `id`, `domain` (`sdd`, `spec`, `code`, `memory`), `requiredTools`, `optionalTools`, `fallbacks`.
- **Provider**: implementación concreta de una o más capabilities, resuelta según `discoveryOrder` y perfil de discovery (`filesystem-first`, `gateway-first`, `local-only`).
- **Adapter**: capa de traducción desde capacidades del producto hacia un runtime objetivo (IDE/CLI externo), expuesta como `generate<Target>Config(args)`.
- **Profile / profile triple**: combinación de `functionalProfile` (`builder` | `product-owner`) + `operationalOverlay` (`local-only` | `standard` | `enterprise`) + `adapterTarget`, persistida en `axiom.yaml` y resuelta a `ResolvedInstallProfile`.
- **Overlay operacional**: modo de riesgo/compliance del proyecto (`local-only`, `standard`, `enterprise`), que determina discovery order, expectativa de gateway y `minimumSignals` de telemetría.
- **Target (adapter target)**: IDE o CLI externo de destino (`opencode`, `claude-code`, `github-copilot`, `vscode`, `cursor`, `litellm`, `copilot-vscode`, `antigravity`, `visual-studio-2026`).
- **Support level**: nivel de soporte de routing por target: `multi-mode` (routing per-slot completo), `single-mode` (routing global, sin per-slot), `fallback-only` (sin routing).
- **Slot**: carril operativo de routing de modelo (`increment`, `bug`, `plan`, `implementation`, `qa-e2e`, `review`, `archive`).
- **ResolvedInstallProfile**: resultado materializado de componer el profile triple contra `profiles.yaml`; input de adapters e installer.
- **Managed state**: estado versionado de `@axiom/versioning` usado para calcular migraciones y checkpoints de `axiom upgrade`.
- **Checkpoint**: snapshot pre-mutación (upgrade, uninstall) restaurable; se conservan los últimos 5.
- **GATE**: verificación explícita y numerada (p. ej. GATE 0010, 0024, 0031) que `@axiom/doctor` puede comprobar en runtime, ligada a un incremento concreto.
- **Spec repository**: fuente compartida del comportamiento y las decisiones del producto (`Axiom.Spec/` para este workspace).
- **Product runtime**: implementación ejecutable ubicada bajo `Axiom/`.
- **Result\<T, E\>**: tipo de retorno sin excepciones usado en todo el dominio (`@axiom/core`) para modelar éxito/fallo explícito.
- **Dogfooding**: construir Axiom usando la propia disciplina SDD de Axiom (modelo de tres repos de este workspace).

## Términos añadidos por el roadmap de rediseño (cerrado)

- **`schemaVersion`**: campo entero presente en todo schema persistido del producto (`axiom.yaml`, registro de usuario, `TopologyManifest`, índices de skills/contexto técnico, `mcp.yml`) que permite evolución aditiva sin romper consumidores de una versión anterior — nunca se resuelve mal una versión como otra (ver NFR-AXM-011 en [02_Requisitos_No_Funcionales.md](02_Requisitos_No_Funcionales.md)).
- **`ArtifactKind`**: tipo cerrado de artefacto en el modelo de carpeta-por-artefacto (`increment | bug | plan | adr | decision`), cada uno con su propio `metadata.yml` y (salvo `adr`/`decision`) su propio vocabulario `WorkflowState`.
- **`TopologyManifest`**: schema versionado (`Axiom/packages/topology/src/types.ts`) que declara el modo de topología de un proyecto (`single-repo` | `multi-repo`), sus repos por rol (`sddRepo`, `specRepo`, `roleCodeRepositories`) y sus asignaciones. Ver [03_Modelo_Operativo_y_Datos.md](03_Modelo_Operativo_y_Datos.md).
- **`RepoRoleKey`**: clave de rol de repo dentro de un proyecto gestionado por Axiom — `sdd` (método/factory), `spec` (conocimiento canónico) o `code` (runtime instalable). No confundir con los roles funcionales del profile triple (`functionalProfile: builder | product-owner`).
- **`allowedWriteScope`**: campo de `PlanMetadata` (`{ repo: string; paths: string[] }[]`) que declara qué paths, en qué repos, un plan tiene permitido mutar; validado en runtime por `validateWriteScope` (ver RF-AXM-019 en [01_Requisitos_Funcionales.md](01_Requisitos_Funcionales.md)).
- **`capabilityId`**: identificador de una herramienta MCP registrada en `@axiom/mcp-tools` (dominios `sdd.*`/`spec.*`), despachada a través del `routeTool` existente de `@axiom/tool-routing` — no debe confundirse con el `id` de `Capability` del modelo de capabilities/providers preexistente (`@axiom/capability-model`), aunque ambos comparten el mismo mecanismo de despacho ortogonal (ver [06_Integraciones_y_Capacidades.md](06_Integraciones_y_Capacidades.md)).
- **`WorkflowId`**: identificador de tipo de workflow (`'increment' | 'bug' | 'plan' | 'role' | 'qa-e2e'`) usado como clave del registro singleton de `workflow-state.json` — un registro por tipo de workflow, no por instancia de artefacto.
- **`externalRefs`**: mecanismo agnóstico de proveedor, disponible en todo tipo de artefacto, para enlazar un artefacto de Axiom con un elemento de un sistema externo (p. ej. Azure DevOps). No debe confundirse con el flag de UI `field.externalRef?: boolean` del plugin de Azure DevOps.
- **`AdrStatus` / `DecisionStatus`**: vocabularios de estado cerrados y no solapados para artefactos `adr` (`proposed | accepted | superseded | rejected`) y `decision` (`proposed | accepted | rejected`, sin `superseded`).
- **`contextBudget`**: parámetro (`small`|`medium`|`large`) de la herramienta MCP `spec.implementationContextRead` que controla el nivel de inlining de contenido en la respuesta, nunca la presencia de campos.

## Términos añadidos por el setup de workspace multi-repo (cerrado, INC-20260705-*)

- **Workspace setup**: la operación (`runWorkspaceSetup`, `apps/cli/src/commands/workspace-setup.ts`) que scaffoldea y cablea un workspace Axiom multi-repo en una sola llamada — repo de control (SDD) + repo de spec + N repos de código por rol — con `axiom.yaml#paths` recíprocos, un `axiom.config/topology.yaml`, bindings locales, registro v2 y config MCP. Ruta aditiva, distinta de `axiom init`/`runInit` (single-repo). Ver [03_Modelo_Operativo_y_Datos.md](03_Modelo_Operativo_y_Datos.md).
- **Repo de control (SDD)**: el repo con `RepoRoleKey` `sdd` dentro de un workspace multi-repo; único repo que recibe los artefactos project-scoped no-por-repo del setup (`axiom.config/topology.yaml`, `.axiom-state/local/topology-bindings.yaml`, `.axiom/mcp.yml`). No confundir con `Axiom.SDD/` (el repo de gobierno de ESTE workspace de desarrollo).
- **Repo de código de rol**: cada repo con rol funcional/de implementación (`backend`/`frontend`/`qa-e2e`/custom, strings abiertos) que el setup de workspace registra en `TopologyManifest.roleCodeRepositories`; recibe solo su propio `axiom.yaml` (con `paths` recíproco) más estado runtime, nunca `topology.yaml`.
- **`topology-bindings.yaml`**: fichero de bindings locales por usuario en `<controlRepo>/.axiom-state/local/topology-bindings.yaml` (`LocalBindings { schemaVersion: 1; localPaths: Record<string, string> }`), que mapea cada `topologyId` a su path absoluto local. Explícitamente no versionado. Escrito por `runWorkspaceSetup` vía `saveLocalBindings` (`@axiom/topology`). Ver [03_Modelo_Operativo_y_Datos.md](03_Modelo_Operativo_y_Datos.md).
- **`mcp.yml`**: config MCP **por proyecto** (`McpProjectConfig`, `schemaVersion: 1`, en `<controlRepo>/.axiom/mcp.yml`) que declara qué procesos servidor MCP están habilitados, en qué scope, para que los adapters generen config runtime. Distinta de `mcp-manifest.yaml` (catálogo de capabilities MCP); no deben fusionarse (ver tabla comparativa en [03_Modelo_Operativo_y_Datos.md](03_Modelo_Operativo_y_Datos.md)). `.axiom/` está gitignoreado, así que es un artefacto machine-local.
- **SDD MCP / Spec MCP**: los dos servidores MCP que el setup de workspace declara en `.axiom/mcp.yml` (INC-20260705-workspace-mcp-generation) — `sdd-mcp-server` (SDD MCP, `scope: 'repo'`, `targetRepo` = clave de registro del repo de control) y `spec-mcp-broker` (Spec MCP, `scope: 'repo'`, `targetRepo` = clave de registro del repo de spec), ambos `enabled: true`. Ver [06_Integraciones_y_Capacidades.md](06_Integraciones_y_Capacidades.md).
- **Adapters (multi-select en el wizard)**: el step `adapters` del wizard de setup de workspace (`INC-20260705-workspace-adapters-multiselect`) que reemplaza al step `target` (select único); un multi-select de los 8 `ADAPTER_TARGETS` (default `['opencode']`) cuya selección se instala en TODOS los repos del workspace y se persiste en `workspace.json`. `spec.target` se conserva como el adapter primario derivado (`adapters[0]`), back-compat. No confundir con **Target (adapter target)** (un IDE/CLI de destino individual). Ver [05_Interfaces_Operativas.md](05_Interfaces_Operativas.md).
- **`workspace.json`**: registro de la selección de adapters del setup de workspace (`WorkspaceSetupRecord`, `schemaVersion: 1`, en `<controlRepo>/.axiom-state/<projectId>/workspace.json`): `{schemaVersion, adapters, profile, overlay, createdAt}`. Escrito una sola vez en el repo de control (`INC-20260705-workspace-adapters-multiselect`). Ver [03_Modelo_Operativo_y_Datos.md](03_Modelo_Operativo_y_Datos.md).
- **skills-catalog / skills-index**: los dos artefactos de la baseline de skills que el setup scaffoldea en el repo de control recién creado (`INC-20260705-workspace-sdd-skills`) — `axiom.config/skills-catalog.yaml` (`SkillsCatalog`, `schemaVersion: 1`, catálogo semilla de 5 ids con `bundleHash` por entrada, obligatorio para el check `TC-010`) y `axiom.config/skills-index/<role>.yaml` (`SkillsRoleIndex` por rol funcional, opcional, check `TC-012`). Versionados y validados de forma independiente (ver RF-AXM-020 en [01_Requisitos_Funcionales.md](01_Requisitos_Funcionales.md)); no confundir el catálogo (fuente por id) con el índice por rol (curación mandatory/available por rol).
- **Base de spec (scaffold)**: la estructura canónica de spec + contexto técnico (`specs/README.md` + `specs/00..08` + `context/TECHNICAL_CONTEXT.md`/`README.md` + directorios estructurales) que el setup scaffoldea en el repo de Spec recién creado desde plantillas bundleadas, guardado per-file sin clobber (`INC-20260705-workspace-spec-base`). Ver [03_Modelo_Operativo_y_Datos.md](03_Modelo_Operativo_y_Datos.md) y [04_Flujos_SDD_y_Ciclo_de_Vida.md](04_Flujos_SDD_y_Ciclo_de_Vida.md).
- **Plantillas de adapter bundleadas**: los templates de instrucciones (`AGENTS_MD_TEMPLATE`/`COPILOT_INSTRUCTIONS_TEMPLATE`, bundleados como constantes TS desde `Axiom.Spec/templates/`) que el setup pasa vía el parámetro opcional `templateContent` a los generadores de opencode/claude-code (y como plantilla a `writeCopilotInstructions`), de modo que esos adapters produzcan ficheros de instrucciones reales en repos recién creados sin degradar a `template-missing` (`INC-20260705-workspace-adapter-templates`). Ver [06_Integraciones_y_Capacidades.md](06_Integraciones_y_Capacidades.md).

## Términos añadidos por la tanda 4 (server MCP ejecutable + e2e, cerrado, INC-20260708-*)

- **`@axiom/mcp-server` / `axiom mcp serve`**: el server MCP **ejecutable** de Axiom (`INC-20260708-mcp-runnable-server`) y el comando CLI que lo lanza (`axiom mcp serve --kind <sdd|spec> --project-root <path> [--home-dir <path>]`). Habla JSON-RPC 2.0 sobre stdio delimitado por saltos de línea (`initialize`/`tools/list`/`tools/call`/`ping`, `-32601` para métodos desconocidos) y expone los handlers de `@axiom/mcp-tools` como herramientas MCP: 7 `sdd.*` con `--kind sdd`, 10 `spec.*` con `--kind spec` (derivadas del campo `domain` de cada handler). Es el proceso real detrás de los identificadores **SDD MCP / Spec MCP** (`sdd-mcp-server`/`spec-mcp-broker`), que antes eran solo declaraciones sin server ejecutable. Ver [06_Integraciones_y_Capacidades.md](06_Integraciones_y_Capacidades.md) y [05_Interfaces_Operativas.md](05_Interfaces_Operativas.md).
- **Launch config de MCP (`command`/`args`)**: los campos opcionales y aditivos (`command?`, `args?`, `env?`) que `McpServerEntry` y las formas `mcp.json` de adapter ganaron en `INC-20260708-mcp-launch-config-wiring`, que describen cómo arrancar el proceso servidor detrás de una entrada MCP. `runWorkspaceSetup` los emite en `sdd-mcp-server`/`spec-mcp-broker` apuntando a `axiom mcp serve` (`command: 'axiom'` + `args: ['mcp','serve','--kind',<sdd|spec>,'--project-root',<path absoluto>]`), en `.axiom/mcp.yml` y en la config MCP nativa por herramienta. Backward-compatible (opcionales). Ver [03_Modelo_Operativo_y_Datos.md](03_Modelo_Operativo_y_Datos.md) y [06_Integraciones_y_Capacidades.md](06_Integraciones_y_Capacidades.md).
- **Config MCP nativa por herramienta**: el fichero de config MCP en el schema NATIVO real que cada herramienta destino lee para auto-lanzar sus servers MCP, emitido por `runWorkspaceSetup` por cada repo × cada adapter seleccionado (`INC-20260708-mcp-native-config-mapping`, `apps/cli/src/commands/native-mcp-config.ts`, `writeNativeMcpConfig`). Un fichero + schema por familia de target: `claude-code` → `.mcp.json` y `cursor` → `.cursor/mcp.json` (`{ mcpServers:{ <id>:{ command, args, env? } } }`); `copilot-vscode`/`github-copilot` → `.vscode/mcp.json` (`{ servers:{ <id>:{ type:'stdio', command, args, env? } } }`); `opencode` → `opencode.json` (`{ $schema, mcp:{ <id>:{ type:'local', command:[cmd,...args], enabled, environment? } } }`). Merge-preserving (solo reemplaza las entradas Axiom-managed por id, preserva el resto; JSON malformado → warning sin clobber) y atómico. `antigravity`/`visual-studio-2026`/`litellm` no tienen schema MCP nativo verificado → warning sin fichero. **Supersede** el antiguo custom-shape `.opencode/mcp.json`/`.claude/mcp.json`, que la ruta de workspace ya no escribe; `.axiom/mcp.yml` permanece como fuente canónica de Axiom. Ver [06_Integraciones_y_Capacidades.md](06_Integraciones_y_Capacidades.md), [05_Interfaces_Operativas.md](05_Interfaces_Operativas.md) y [03_Modelo_Operativo_y_Datos.md](03_Modelo_Operativo_y_Datos.md).
- **Aislamiento MCP por proyecto (enforced)**: garantía de que el server MCP ejecutable (`@axiom/mcp-server`) solo lee/opera sobre el proyecto al que está atado al arrancar, **hecha cumplir por código** y no solo por convención (`INC-20260708-mcp-project-isolation-hardening`). Todo campo que identifica proyecto en los argumentos de `tools/call` (`projectId`/`homeDir`/`projectRoot`/`specScopeAbsolutePath`/`sddScopeAbsolutePath`) se fija (pin) al `context` del server; un valor del llamador que difiera devuelve `isError: true` con `"Cross-project access blocked: this MCP server is scoped to project '<id>'. Launch the MCP server for the target project instead."`, y `sdd.projectRegistryRead` se acota a la entrada del proyecto propio (nunca el registro machine-wide). Mismo patrón "bind at construction, reject on mismatch" que `@axiom/memory` (GATE 0024). Los selectores dentro-de-proyecto siguen siendo caller-supplied. Ver [07_Gobierno_y_Seguridad.md](07_Gobierno_y_Seguridad.md) y [06_Integraciones_y_Capacidades.md](06_Integraciones_y_Capacidades.md).
- **/axiom-autopilot (orquestación desatendida)**: comando + skill de Claude Code (`.claude/commands/axiom-autopilot.md` + `.claude/skills/axiom-autopilot.md`, `INC-20260708-axiom-autopilot-command`) que codifica el playbook de orquestación multi-incremento desatendida (descomponer una tanda → subagentes `axiom-increment` secuenciales → auto-responder ambigüedad → verificación independiente → integrar en `00–08` → archivar → resumen de decisiones; nunca git mutante, nunca detenerse a preguntar). Es tooling de DESARROLLO para construir Axiom (vive en `.claude/`), no una capacidad de runtime del producto. Ver [04_Flujos_SDD_y_Ciclo_de_Vida.md](04_Flujos_SDD_y_Ciclo_de_Vida.md).

## Términos añadidos por la tanda INC-20260709-* (review ledger, recall, terse single-source, engram stdio nativo)

- **Ledger de hallazgos (review findings ledger)**: registro estructurado que el flujo de revisión emite y persiste (`INC-20260709-review-findings-ledger`), con campos `id` (`{LENS}-{NNN}`), `lens`, `location` (`file:line`), `severity` (BLOCKER|CRITICAL|WARNING|SUGGESTION), `status` (open|fixed|verified|wont-fix|info) y `evidence`. Producto de una primera pasada exhaustiva **loop-until-dry** (barrer hasta N=2 barridos secos consecutivos, techo 4; re-review N=1) y consumido por una **re-review scoped** (verifica los hallazgos del ledger + revisa solo líneas tocadas por el fix; un hallazgo en línea no tocada es `status: info` y no dispara ronda nueva). El contrato se bundlea como constante TS única (`@axiom/document-bootstrap`'s `REVIEW_LEDGER_CONTRACT`) con test de drift. Ver [04_Flujos_SDD_y_Ciclo_de_Vida.md](04_Flujos_SDD_y_Ciclo_de_Vida.md) y [05_Interfaces_Operativas.md](05_Interfaces_Operativas.md).
- **Entrada MCP nativa de engram (stdio, sin `engram serve`)**: entrada `engram` LOCAL de stdio (`engram mcp --project <projectId> --tools agent`) que la config MCP nativa por herramienta añade cuando el provider `engram` está habilitado (`INC-20260709-engram-mcp-stdio-native-config`) — en `mcpServers`/`servers`+`type:'stdio'`/`mcp`+`type:'local'` según el tool. Elimina la necesidad del daemon HTTP `engram serve` (puerto 7437) para los adapters; ninguna config generada referencia engram por HTTP/`ENGRAM_URL`. Nunca se escribe en `.axiom/mcp.yml`. Ver [06_Integraciones_y_Capacidades.md](06_Integraciones_y_Capacidades.md).
