## Open questions

- **Q-legacy-1**: Should the command attempt any status inference from
  legacy body text (e.g. a line like `Status: closed` or `**Done**`), or
  always force every migrated artifact to `status: 'draft'` /
  `'proposed'` regardless of what the legacy file claims? This audit
  recommends: attempt a best-effort, narrow literal match (a line
  matching `^Status:\s*(\w[\w-]*)` mapped through a small known-synonym
  table, e.g. `done|complete -> closed`), but if no match is found or
  the matched value doesn't map to a known `WorkflowState`, default to
  `'draft'` — never silently trust an unparseable or absent status as
  `'closed'`. This is a judgment call for the next role to confirm, not
  blocking.
- **Q-legacy-2**: How should the tool decide legacy artifact *kind*
  (increment vs. bug vs. plan vs. adr/decision) — purely by source
  subfolder name (`increments/`, `bugs/`, `plans/`, `adr/`,
  `decisions/`), or also by content sniffing? This audit recommends
  folder-name-only (mechanical, zero-judgment, mirrors how
  `listArtifacts` itself is folder-keyed) — a legacy repo's own folder
  naming is the only signal that costs nothing to get wrong safely
  (worst case: an artifact lands in the wrong kind and a human corrects
  it during the mandatory review step).
- **Q-legacy-3**: Should the command support legacy repos where
  increments/bugs/plans are single flat `.md` files (e.g.
  `increments/INC-042-foo.md`) as well as the folder-per-artifact shape
  (e.g. `increments/INC-042-foo/README.md` or `.../spec.md`)? Both
  shapes are plausible for "an older SDD repo" and neither is specified
  authoritatively. This audit recommends supporting both: scan each
  kind-folder's immediate children, and for each child that is a
  directory, look for a markdown file inside (preferring `README.md`,
  falling back to the first `.md` found); for each child that is itself
  a `.md` file, read it directly. Not blocking; the next role should
  confirm this covers realistic cases without over-engineering.
- **Q-legacy-4**: What happens on a legacy title collision (two legacy
  files resolving to the same generated Axiom ID, or a legacy file
  whose derived slug collides with an already-existing Axiom artifact)?
  This audit recommends: rely on `generateUniqueArtifactId`'s existing
  uniqueness guarantee (it already probes for collisions when creating
  new artifacts) and skip-with-warning rather than overwrite if an
  identical target somehow already exists — never silently overwrite an
  existing Axiom artifact. Not blocking; next role's implementation
  detail.

## Assumptions

- "Legacy SDD format" (the migration's INPUT) is defined for this
  increment as: a directory tree with zero or more of these
  loosely-named top-level subfolders — `increments/`, `bugs/`, `plans/`,
  `adr/`, `decisions/` (matching the exact folder names this
  workspace's own `Axiom.SDD/AGENTS.md` names for increments/bugs, plus
  the `adr`/`decisions` naming INC-11 already established for the
  *target* Axiom format, reused here as the *source* naming assumption
  too since no distinct legacy naming is specified anywhere). Each
  subfolder contains per-artifact entries, each either (a) a
  subdirectory containing a markdown file (preferring `README.md`), or
  (b) a single top-level `.md` file. A title is extracted from the
  first `# heading` line found; a best-effort `status` is extracted per
  Q-legacy-1; everything else in the file body is carried into the new
  Axiom artifact's content area verbatim (not restructured into
  goal/scope/acceptance-criteria sections — that restructuring would
  require semantic understanding of arbitrary legacy prose, explicitly
  out of scope). Folders not present are simply skipped (no error) —
  mirrors `listArtifacts`'s own "absent kind folder is not an error"
  behavior.
- This input assumption is **not** a formally specified legacy schema
  because none exists to target: both source docs describe Via B only
  in one sentence ("migra un repo SDD/spec anterior al formato Axiom")
  plus a subagent name list, with no worked example, no field list, and
  no sample legacy file shown anywhere (unlike, say, INC-10's
  `TechnicalContextIndex`, which had a full worked YAML fixture in the
  source doc). This makes the above assumption this audit's own
  invention, grounded in the only concrete "loose lightweight SDD"
  examples actually visible anywhere in this workspace context
  (`Axiom.SDD/AGENTS.md`'s own `increments/`/`bugs/` folder-of-markdown
  convention) — not a literal transcription of a specified format.
- Migrated artifacts are created via the exact same primitives a human
  would use via `axiom increment create` / `axiom-adr create` etc. —
  i.e. this increment does not invent a second, parallel artifact
  creation code path; it is real Axiom commands' machinery, invoked
  programmatically with content sourced from disk instead of from a
  user prompt, per source doc 15.4's explicit rule.
- `Axiom/packages/document-bootstrap`'s `writeGuardedFile` primitive
  (already extracted and reused by INC-18) is NOT the right write
  primitive here, because migrated artifacts are structured
  `metadata.yml` + content artifacts (the `artifact-store.ts`/
  `listArtifacts` folder-per-artifact convention), not arbitrary
  technical-context markdown files with no metadata sidecar. The two
  bootstrap vias reuse two different, already-existing write primitives
  — this is a deliberate distinction, not an inconsistency between
  INC-18 and INC-19.

### Legacy-format input-assumption decision, and honest speculativeness assessment

**This increment is more speculative than most others closed in this
roadmap, and that should be stated plainly rather than hidden behind
confident-sounding scope language.**

Every other closed increment in this chain (INC-06 through INC-18) had
either (a) a concrete existing source-doc worked example/schema to
target (e.g. INC-10's `TechnicalContextIndex` YAML fixture, INC-15's
`mcp.yml` schema with a literal example), or (b) a concrete existing
codebase artifact to audit and extend (e.g. INC-11 extending the
already-shipped `artifact-store.ts`). Via B has neither: source doc
15.3 describes it in one sentence plus a subagent name list, with zero
worked example of what a "repo SDD/spec anterior" actually looks like
on disk. Any input-format decision made here is this audit's own
invention, not a transcription of a specified target.

Given that, the honest, minimal-viable response is **not** to design a
best-effort auto-detecting importer that tries to handle many possible
legacy shapes (that would compound speculation on speculation — guessing
at formats to guess at). Instead: **anchor the input assumption to the
one convention this workspace can actually point to** — `Axiom.SDD/
AGENTS.md`'s own stated (if not literally used-by-this-session)
`increments/`/`bugs/` folder-of-markdown convention, extended
symmetrically to `plans/`/`adr/`/`decisions/` using INC-06/INC-11's own
already-established target-side folder names — and build the smallest
possible mechanical importer against exactly that one assumed shape,
explicitly documented as an assumption, not a spec. See "Assumptions"
above for the exact shape.

**Recommended minimal viable scope**, restated concretely: a single
`axiom bootstrap from-legacy-sdd <path> [--dry-run]` command that:

1. Scans `<path>/{increments,bugs,plans,adr,decisions}/` for immediate
   children (skipping any kind-folder that doesn't exist — not an
   error).
2. For each child, extracts a title (first `# heading`) and a
   best-effort status (Q-legacy-1), reading either the child directory's
   markdown file or the child `.md` file directly (Q-legacy-3).
3. Creates a NEW Axiom-format artifact via the existing creation
   primitives (`generateUniqueArtifactId` + the matching
   `makeInitial*Metadata` factory + `saveArtifactMetadata` +
   `ensureArtifactReadme` — or, more precisely, the equivalent CLI
   `create` subcommand functions already wired to these primitives, per
   source doc 15.4's "structural creation must go through Axiom
   commands, not ad-hoc" rule), carrying the legacy body content into
   the new artifact's content area.
4. Marks every migrated artifact `draft`/`proposed` (never trusts a
   legacy "closed" claim into an Axiom `closed` status without a human
   confirming — see Q-legacy-1), per addendum section 15.
5. `--dry-run` reports what would be created (kind, derived title, derived
   ID, source path) without writing anything — mirrors
   `axiom upgrade --dry-run`'s existing precedent (already used as
   precedent by INC-18's own CLI surface design).

If this minimal scope is still judged too large once a next role starts
actual implementation, the next-smaller viable slice is: implement only
`increments/` and `bugs/` migration first (the two kinds this
workspace's own `AGENTS.md` explicitly names), deferring `plans/`,
`adr/`, `decisions/` migration to a follow-up pass — but this audit
does not recommend pre-emptively cutting that scope now, since the
per-kind mechanical cost is identical (same primitive, different
factory function) and the roadmap's own INC-19 dependency list already
includes INC-11 (ADR/Decision), signaling those kinds were intended to
be in scope from the start.

### Reusability confirmation: `artifact-store.ts` (INC-06) + ADR/Decision (INC-11)

Read `Axiom/packages/workflow/src/artifact-store.ts` directly (relevant
sections: `loadArtifactMetadata`, `saveArtifactMetadata`,
`ensureArtifactReadme`, `listArtifacts`, and all five
`makeInitial*Metadata` factories) and confirmed the real exported name
is `generateUniqueArtifactId` (`Axiom/packages/workflow/src/
artifact-id.ts`) — not `generateArtifactId` as loosely referenced in

**Finding: confirmed genuinely reusable, no new writer needed.**

  built-in collision-avoidance (addresses Q-legacy-4).
- `saveArtifactMetadata` (atomic tmp+rename write of `metadata.yml`) and
  `ensureArtifactReadme` (creates `README.md` only if absent, never
  clobbers) are the exact two write operations a migrated artifact

- `makeInitialIncrementOrBugMetadata`, `makeInitialPlanMetadata`,
  `makeInitialAdrMetadata`, `makeInitialDecisionMetadata` (the last two
  landed by INC-11) cover all five target kinds this migration needs —
  every one of them accepts a caller-supplied `title` and (for
  increment/bug/plan) `status: WorkflowState`, which is exactly the two
  fields this migration's mechanical extraction step produces. No
  factory needs modification; no sixth kind exists to migrate.
- This confirms the task framing's core premise directly: "migrating an
  increment/bug/plan/ADR" **is** "create a new Axiom artifact with

- No modification to `artifact-store.ts` is needed for this migration
  to work — this is a pure consumer of already-shipped, already-tested
  functions, same posture as every other CLI command that creates
  artifacts today.

### Collapsed-role recommendation

**Recommendation: collapse source doc 15.3's suggested 7-subagent chain
(`legacy-spec-reader`, `increment-migrator`, `bug-migrator`,
`plan-migrator`, `context-normalizer`, `adr-extractor`,
`axiom-format-writer`) and the roadmap's own already-partially-collapsed
5-role sequence (`bootstrap-analyzer` -> `migration-engineer` ->
`docs-skills-writer` -> `cli-implementer` -> `validator-reviewer`) down
to 2 roles for this minimal scope: one `cli-implementer` (read legacy
files + extract title/status + create Axiom artifacts via existing
primitives + implement `--dry-run`, in one pass) followed by
`validator-reviewer` (independent review, unchanged).**

Reasoning, mirroring INC-18's own collapse logic applied to this
narrower, even-more-mechanical task:

- Source doc 15.3's 4 per-kind "migrator" subagents
  (`increment-migrator`, `bug-migrator`, `plan-migrator`, plus
  `adr-extractor` for the 5th kind) would each perform the **identical**
  operation — read one file, extract title, extract status,
  call one factory function, save — differing only in which
  `makeInitial*Metadata` factory and which target subfolder name they
  use. This is a single parametrized loop over 5 kinds, not 4-5
  independent judgment calls; splitting them into separate subagents
  would mean serializing/deserializing the same trivial
  `{kind, title, status, body}` shape 5 times across role boundaries for
  zero quality benefit.
- `legacy-spec-reader` (read + parse legacy files) and
  `axiom-format-writer` (create new Axiom artifacts) are, at this
  minimal scope, the two ends of the exact same mechanical pipeline as
  the per-kind migrators above — reading a file and writing a new
  artifact from its extracted fields is one operation, not two
  independent judgment calls, for the same reason INC-18 collapsed
  analysis+drafting into one `bootstrap-implementer` step.
- `context-normalizer` (source doc's suggested role for reconciling
  content style/structure across migrated documents) is explicitly
  **not needed** at this minimal scope, because this audit's
  recommended scope carries legacy body content verbatim rather than
  attempting to restructure/normalize it (see "Non-goals" — deeper
  content normalization is out of scope). If a future increment adds
  real content restructuring, re-introducing a distinct normalization
  role would be justified then, not now.
- `adr-extractor`'s only distinct-from-the-others judgment call in
  source doc 15.3 would be *inferring* ADRs from unstructured legacy
  prose that isn't already in an `adr`/`decisions`-shaped folder (a
  genuine semantic-understanding task). This audit's recommended scope
  does not attempt that inference (see "Non-goals") — it only migrates
  ADRs/Decisions that already live in an explicitly-named legacy
  `adr`/`decisions` folder, which is exactly as mechanical as migrating
  an increment. The one task that would justify a separate
  `adr-extractor` role is precisely the one this scope excludes.
- The roadmap's own already-collapsed 5-role sequence
  (`bootstrap-analyzer -> migration-engineer -> docs-skills-writer ->
  cli-implementer -> validator-reviewer`) still separates
  analysis/reading (`bootstrap-analyzer`) from writing
  (`cli-implementer`) and inserts a `docs-skills-writer` normalization
  step this audit judges unnecessary at minimal scope (see above). This
  audit's own migration-engineer role (this document) already performed
  the "audit legacy shape + confirm reuse" work the roadmap assigned to
  a combination of `bootstrap-analyzer`/`migration-engineer`, so what
  remains for a next role is purely mechanical implementation — one
  `cli-implementer` pass covers it.
- `validator-reviewer` remains distinct and unchanged, consistent with
  every prior increment in this chain and the parent orchestrator's
  "fresh review rule": independent verification of migrated output
  (correct kind/ID mapping, all-draft status, no silent overwrite, valid
  metadata shape) benefits from being a separate, non-authoring pass.
- Like INC-18's own collapse, this is scoped to the stated minimal
  input assumption and Non-goals above. If a future increment adds real
  ADR inference from unstructured prose or content restructuring,
  re-splitting into more roles would likely be justified then.

### CLI surface design

**Recommendation: `axiom bootstrap from-legacy-sdd <path> [--dry-run]`**,
extending the `bootstrap` namespace INC-18 already established
(`axiom bootstrap from-code --level minimal|basic`), per the existing
`axiom <namespace> <verb>` convention and per `bootstrap.ts`'s own
header comment already reserving this exact sibling-verb slot.

- **Namespace**: `bootstrap` (existing, from INC-18 — not a new
  namespace; `registerBootstrap` in `Axiom/apps/cli/src/commands/
  bootstrap.ts` already registers the parent `bootstrap` command and
  its `from-code` subcommand; `from-legacy-sdd` is a second subcommand
  under the same parent).
- **Verb**: `from-legacy-sdd`, mirroring source doc 15.3's own literal
  name for Via B verbatim (same pattern as INC-18's `from-code` mirroring
  Via A's literal name).
- **Positional argument**: `<path>` — the filesystem path to the legacy
  SDD/spec repo root (the directory expected to contain
  `increments/`/`bugs/`/`plans/`/`adr/`/`decisions/` subfolders per the
  input assumption above). Required, no default — unlike
  `from-code`'s implicit "analyze the resolved project", Via B
  necessarily reads from a *different*, externally-specified repo
  (the current project's `axiom bootstrap from-legacy-sdd` target and
  its Axiom-format destination are two different repos by definition).
- **`--dry-run`**: reports what would be created (kind, derived title,
  derived ID, source path) without writing anything, mirroring
  `axiom upgrade --dry-run`'s existing precedent (already flagged as a
  reasonable precedent by INC-18's own CLI design section, left
  undecided there, and now concretely recommended here since Via B's
  migration is a one-shot, harder-to-undo bulk-create operation where a
  preview is more valuable than for Via A's idempotent-ish draft
  regeneration).
- **Target resolution**: the destination Axiom project is the
  already-resolved current project (same `resolveProject`/`specScope`
  pattern `from-code` uses) — `<path>` names only the *source* legacy
  repo, never the destination. This keeps the command symmetric with
  `from-code`'s existing resolution pattern for "where do I write,"
  differing only in "where do I read from."
  Confirming the exact resolution call (and what happens if no Axiom
  project is resolved at the current cwd — almost certainly the same
  `exitCode: 1` + "ejecutá axiom init" guidance `from-code` already
  gives) is the next role's (`cli-implementer`'s) task, not decided
  here.
- **Output contract**: mirrors `from-code`'s existing convention —
  report which artifacts were created (kind, ID, title, source path)
  and print an explicit reminder that migrated content is
  `draft`/`proposed` and requires human review before relying on or
  approving it, per addendum section 15's explicit callout of
  "migraciones desde SDD legacy" as one of the named categories
  requiring review.
- Not designed here (left to the next role): exact flag list beyond
