# Increment: plan-role-split (P1-5)

Status: closed
Date: 2026-07-10

## Goal

Make `axiom-plan create` actually produce a plan whose `metadata.yml`
separates work by role — a core premise of the plan artifact that was
never implemented. The plan-metadata template
(`plan-metadata-template.yaml#roles`) already defined the shape
(`required`/`roleFiles[]`), but the runtime `PlanMetadata`
(`@axiom/workflow`'s `artifact-store.ts`) had no `roles` field at all,
and `makeInitialPlanMetadata` always left `targetRepos`/
`allowedWriteScope` empty — so `axiom validate changes` had nothing
real to enforce.

## Context

Builds directly on `INC-20260710-dynamic-team-roles` (Decision D5),
which introduced `topology.yaml#roles` (a first-class `RoleDef[]`
registry) and `#assignments` (repo↔role mapping). This increment wires
the plan artifact's role-split to that DYNAMIC registry — never a
fixed role list — via `@axiom/topology`'s `loadTopology`.

## Scope

- Extend `PlanMetadata` (`packages/workflow/src/artifact-store.ts`)
  with an OPTIONAL `roles: { required: string[]; roleFiles: {...}[] }`
  field, matching `plan-metadata-template.yaml#roles`'s shape
  verbatim. Shape-guard + parse/serialize it in `loadArtifactMetadata`/
  `saveArtifactMetadata`; extend `makeInitialPlanMetadata`.
- `axiom-plan create` (`apps/cli/src/commands/axiom-plan.ts`):
  - New `--roles <csv>` option (explicit role id list).
  - When `--roles` is omitted, derive the role list from
    `topology.yaml#roles`/`#assignments` (`derivePlanRoles` helper).
  - For each role: emit a `roleFiles` entry (`role-<slug>.md`,
    `status: queued`) and write a thin `role-<slug>.md` stub file next
    to `metadata.yml` (single flat file per role — no deep
    autogenerated hierarchies, per `AGENTS.md`'s bootstrap limits).
  - Populate `targetRepos`/`allowedWriteScope` from the roles' assigned
    repos (via `topology.yaml#assignments`) UNLESS the caller passed
    its own `--target-repo` (which still wins, unchanged behavior).
  - Missing topology / no registered team roles degrades gracefully
    to an empty role-split (`roles: {required: [], roleFiles: []}`) +
    a clear note in the result message — never crashes.
- Verified (added tests, no code change needed) that `axiom validate
  changes` / `validateWriteScope` already consume `allowedWriteScope`
  generically — the only real gap was that `axiom-plan create` never
  populated it. A role-split plan created via `create` now has a
  populated `allowedWriteScope` that `validate changes` enforces
  end-to-end (proven with a new integration test).

## Non-goals

- Not implementing the full richer `plan-metadata-template.yaml`
  schema (`riskModel`, `gates`, `verification`, `integration`,
  `executionTrace`) — only its `roles` section, which is what the
  runtime `PlanMetadata` needed.
- Not adding per-role sub-path scoping within a repo — a role owns the
  WHOLE repo(s) it's assigned to in the topology model (topology has
  no notion of sub-paths per role), so `allowedWriteScope` uses a
  single `'**'` glob per assigned repo.
- Not adding a `--id <planId>` override to `axiom-plan create` (plan
  IDs stay system-generated, unchanged from before this increment).

## Acceptance criteria

- [x] `PlanMetadata` has an optional `roles` field matching the
      template shape; existing plans without it still parse
      (back-compat verified by test).
- [x] `makeInitialPlanMetadata` accepts `roles`/`targetRepos`/
      `allowedWriteScope` and emits them.
- [x] `axiom-plan create --roles <csv>` and the topology-derived
      default both populate `roles.roleFiles`, `targetRepos`, and
      `allowedWriteScope`, and write one `role-<slug>.md` per role.
- [x] Missing topology/roles degrades gracefully (no crash, empty
      role-split + note).
- [x] `axiom validate changes` enforces the populated
      `allowedWriteScope` of a role-split plan (integration test).
- [x] `npm run build` exits 0; `npm test` is green (207 files / 2193
      tests, +9 vs. the 2184 baseline).

## Open questions

None blocking. `--id <planId>` override for `create` is a possible
future addition but out of scope here (not requested explicitly).

## Assumptions

- A team role in `topology.yaml#roles` maps to whole repo(s) via
  `#assignments`, not sub-paths — so `'**'` is the correct default
  scope glob per assigned repo, not a placeholder to refine later.
- `roles.required` is populated with the same role id list used to
  build `roleFiles` (every role included in a plan's role-split is
  "required" for that plan; there's no notion of an optional role in
  the MVP).

## Validation

- `npm run build` (from `Axiom/`): exit 0, no errors.
- `npm test` (from `Axiom/`): 207 test files / 2193 tests passed (was
  207 files / 2184 tests before this increment; +9 new tests, 0
  failures, 0 regressions).
- Live proof in a scratchpad temp project (cleaned up after): seeded
  `axiom.config/topology.yaml` with `roleCodeRepositories` `api-repo`/
  `web-repo`; ran `axiom roles register backend --repo api-repo` and
  `axiom roles register frontend --repo web-repo`; ran
  `axiom-plan create --title "Role probe plan" --roles backend,frontend
  --json` — the emitted `metadata.yml` has:
  ```yaml
  targetRepos: [api-repo, web-repo]
  allowedWriteScope:
    - repo: api-repo
      paths: ['**']
    - repo: web-repo
      paths: ['**']
  roles:
    required: [backend, frontend]
    roleFiles:
      - role: backend
        slug: backend
        file: role-backend.md
        status: queued
      - role: frontend
        slug: frontend
        file: role-frontend.md
        status: queued
  ```
  and `role-backend.md`/`role-frontend.md` were created in the plan
  folder. Re-ran `axiom-plan create` WITHOUT `--roles` (topology-only
  auto-derivation) and got the identical role-split, confirming both
  paths work.

## Result

Plans created via `axiom-plan create` now genuinely separate work by
role, derived from the project's dynamic team-role registry
(`topology.yaml#roles`/`#assignments`), with a real, enforceable
`allowedWriteScope` per assigned repo. `axiom validate changes` was
already generic enough to consume it — the fix closes the actual gap
(an always-empty `PlanMetadata` at creation time), not the validation
wiring itself.
