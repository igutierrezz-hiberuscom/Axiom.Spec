### Explicit scope-minimization judgment call

`Axiom.SDD/AGENTS.md`'s "Explicit Bootstrap Limits" section prohibits
"heavy multi-agent orchestration frameworks" and "deep autogenerated
folder hierarchies" unless explicitly requested. Source doc 15.3's literal
7-subagent chain (`repo-analyzer`, `backend-analyzer`, `frontend-analyzer`,
`qa-analyzer`, `architecture-summarizer`, `context-writer`,
`index-builder`) is exactly that kind of construct if implemented
literally as 7 distinct orchestrated roles/prompts/handoff files.

The user's request to execute this roadmap sequentially is a legitimate
basis to build **some** version of bootstrap-from-code — it is explicitly
item 18 of an already-agreed plan, not a speculative addition. But
"some version" does not mean "the literal 7-role chain." This audit's
judgment: the actual need — generate initial technical-context/skills
draft content from an existing codebase, output as draft/review, never
auto-approved, written through the existing writer/command surface — is
satisfiable by **one mechanical analysis+draft+write pass** at the two
cheapest exhaustiveness levels (see "Exhaustiveness-level scope decision"
below), not by 7 distinct subagent roles. See "Collapsed-role
recommendation" below for the reasoning grounded in what Level 0/1 work
actually requires.

## Open questions

- **Q-bootstrap-1**: Should Level 0/1 bootstrap output populate
  `TechnicalContextIndex.available` entries only, or also
  `mandatory.always`? This audit recommends `available`-only for an
  automated first pass (a bootstrap tool should not unilaterally declare
  documents "mandatory reading" for a role without human confirmation —
  that is a curation judgment, not a mechanical one); `mandatory.*`
  population should require an explicit human promotion step (e.g. a
  future `axiom bootstrap promote` or manual edit), consistent with
  addendum section 15's review requirement. Not blocking; the next role
  (implementer) should confirm before writing the index-writing logic.
- **Q-bootstrap-2**: Should the drafted markdown documents carry an
  explicit in-file marker (e.g. a `<!-- AXIOM:DRAFT -->` banner, mirroring
  `document-bootstrap`'s existing `AXIOM:GENERATED`/`TEAM:CUSTOM` marker
  convention) so a human reviewer can immediately see a file was
  machine-drafted, distinct from the index-level draft/review status?
  This audit recommends yes, for defense-in-depth (a document could be
  copied out of its index context and lose the "needs review" signal
  otherwise), but leaves exact marker text to the next role.
- **Q-bootstrap-3**: Where does "draft/review state" actually live for a
  bootstrap-generated `TechnicalContextIndex`, given the index schema
  (INC-10) has no `status` field today (unlike `WorkflowState`'s
  `draft|...` vocabulary used by increment/bug/plan artifacts)? Options:
  (a) add an optional `status?: 'draft' | 'reviewed'` field to
  `TechnicalContextIndex` as a schema-writer follow-up, (b) track review
  status out-of-band (e.g. a sibling `.review-status.yml` or a checklist
  in the generated guide document), (c) rely entirely on the fact that
  the files are new/untracked in git until a human commits them (weakest
  option, easiest to bypass accidentally). Not blocking this audit; flagged
  for the next role's design step. This audit leans toward (a) if INC-10's
  schema is still cheap to extend additively, deferring to a schema-writer
  role to confirm.

### `writer.ts` reusability confirmation

Read `Axiom/packages/document-bootstrap/src/writer.ts` in full (165
lines) plus its companion `types.ts` (132 lines).

**Finding: the write *primitive* (path-guard -> atomic write) is
genuinely reusable in principle; the concrete exported function,
`writeCopilotInstructions`, is not reusable as-is.**

Confirmed by direct read:

- `writeCopilotInstructions(opts: WriteOpts)` is hardwired end-to-end to
  one specific artifact: a single Markdown file at a single default path
  (`DEFAULT_TARGET_REL = '.github/copilot-instructions.md'`), rendered
  from one specific renderer (`renderCopilotInstructions`) driven by one
  specific input shape (`ResolvedVariables`, 5 required fields:
  `project.name`, `project.id`, `mcp.allowlist`, `support.level`,
  `target.id`). `WriteResult.writtenFiles` is even hardcoded to
  `[DEFAULT_TARGET_REL]` regardless of `opts.targetPath` (line 161) — a
  latent inaccuracy for any caller that overrides the target path, but
  irrelevant to today's single call site.
- The function performs four genuinely reusable steps, in order: (1)
  **path-guard** (lines 53-69: resolve `targetPath`, reject if it
  escapes `projectRoot` via `path.relative` + `..`/absolute checks,
  *before* any I/O — `AD-10`); (2) **classify + preserve**
  (`classifyAndPreserve`, idempotency.ts: splits existing file content
  into preamble/`AXIOM:GENERATED`/`TEAM:CUSTOM` regions so
  human-added content survives regeneration); (3) **merge** (rebuilds
  `preamble + newAxiomBlock + newTail` with documented precedence:
  existing team block > template team block > default); (4) **atomic
  write** (lines 145-158: `mkdirSync(recursive)` + write to `.tmp` +
  `renameSync`).
- Of these four, **(1) path-guard and (4) atomic write are directly,
  literally reusable** for writing new technical-context documents — they
  have no dependency on `ResolvedVariables`, the copilot-instructions
  renderer, or the single hardcoded target path; they are pure
  path/filesystem operations. **(2) classify-and-preserve and (3) merge**
  are reusable *as a pattern* only if a bootstrap-drafted document is
  expected to be re-run and merged against human edits the same way
  `AGENTS.md` is (plausible and desirable per Q-bootstrap-2's marker
  idea, but not yet a stated requirement for a first bootstrap pass,
  where files are typically being created for the first time, not
  re-merged).
- **Conclusion**: bootstrap-from-code's writer should **extract and reuse
  the path-guard + atomic-write primitive** (steps 1 and 4) as a small,
  generic helper — e.g. lift them into a shared function taking
  `(projectRoot, targetPath, content)` — rather than calling
  `writeCopilotInstructions` itself, which cannot write more than one
  file at one fixed path from one fixed variable shape. This is a
  **minor, additive refactor** (extract two already-pure blocks of
  existing, tested code into a reusable helper), not new I/O logic, and
  satisfies the roadmap's own diff statement ("the rendering/writing
  tail-end already exists in `writer.ts` and should be reused for the
  final write step") accurately — it is the *primitive*, not the
  *function signature*, that is reused. The team-block-preservation
  behavior remains valuable and should still be offered as an *opt-in*
  merge mode for re-running bootstrap against files a human has since
  edited, once that scenario is actually needed (not required for this
  increment's Level 0/1 scope, where a bootstrap run onto a project with
  no prior technical-context files is the primary case).

### `TechnicalContextIndex`/`loadTechnicalContextIndex` fitness confirmation

Read `Axiom/packages/technical-context/src/technical-context-index.ts`
in full (514 lines) plus the worked fixture
(`tests/fixtures/technical-context/indexes/backend.index.yml`).

**Finding: confirmed fit, with one deliberate constraint that shapes the
bootstrap step's design.**

- `TechnicalContextIndex` (`{schemaVersion: 1, projectId?, role,
  repoKinds, mandatory: {always, whenTags}, available}`) is exactly the
  artifact a bootstrap step should populate for its "technical-context
  indexes" deliverable (source doc 15.1). `available` entries
  (`{id, path, summary?, tags}`) map directly onto what a mechanical
  Level 0/1 pass can produce: a `path` to a drafted document, an `id`,
  and a best-effort `summary`/`tags` derived from repo structure (e.g.
  detected stack, folder names) — no deep semantic understanding
  required.
- **Constraint**: every document `path` field is **required**, and
  `loadTechnicalContextIndex`/`validateTechnicalContextIndex` perform
  **shape validation only** — they do not verify the referenced `path`
  resolves to a file that exists on disk (confirmed by
  `general-spec.md`'s "Technical-context index" section and by reading
  the validator: no `fs.existsSync` check on any `path` field anywhere in
  `technical-context-index.ts`). This means a bootstrap step's index and
  its drafted documents are two independent write operations with no
  built-in referential-integrity check between them — the bootstrap
  implementer must write both the documents and the index entries
  referencing them in the same pass and keep the paths consistent by
  construction; nothing downstream will catch a drift between them today.
  This is a pre-existing, documented gap (not introduced by this
  increment) and out of scope to fix here.
- `technicalContextIndexPath(specScopeAbsolutePath, roleOrKind)` gives the
  exact target path convention
  (`<specScopeAbsolutePath>/technical-context/indexes/<role-or-kind>.index.yml`)
  a bootstrap step must write to, and the drafted document paths
  (`architecture.md`, `commands.md`, etc.) are conventionally siblings
  one level up (`../backend/architecture.md` in the worked fixture) —
  i.e. `<specScopeAbsolutePath>/technical-context/<role-or-kind>/*.md`.
  This gives the bootstrap step a concrete, already-established path
  convention to target, not a new one to invent.

### Exhaustiveness-level scope decision

**Recommendation: build only Level 0 (Minimal) and Level 1 (Basic) in
this pass. Explicitly defer Level 2 (Standard), Level 3 (Deep), and
Level 4 (Exhaustive) to a future increment.**

Reasoning, grounded in what each level actually requires (source doc
15.2) rather than a blanket minimization stance:

- **Level 0 (Minimal — "solo estructura y placeholders")** and **Level 1
  (Basic — "resumen general, mapa de repos, stack detectado, comandos
  principales")** are both **mechanical, low-AI-judgment operations**:
  detecting a stack (package manifests, lockfiles, config files),
  enumerating repo/folder structure, extracting declared build/test/lint
  commands (`package.json` scripts, `Makefile` targets, CI config), and
  writing placeholder or templated-summary documents plus `available`
  index entries pointing at them. None of this requires understanding
  business logic, architecture intent, or code semantics — it is closer
  to `axiom doctor`'s existing evidence-gathering style (read files,
  report facts) than to genuine AI analysis.
- **Level 2 (Standard)** already asks for "contexto técnico inicial por
  rol, arquitectura por repo, patrones principales, ejemplos básicos" —
  this requires actually reading and understanding code to identify
  architectural patterns and produce representative examples, a
  qualitatively different (and much more expensive, much more
  review-dependent) task than Level 0/1's structural detection.
- **Level 3 (Deep)** and **Level 4 (Exhaustive)** explicitly ask for
  cross-repo flow analysis, business-rule detection, entity/endpoint
  inventories, and traceability from old specs — genuine deep semantic
  understanding of a codebase, with correspondingly higher AI cost and
  human-review burden exactly as source doc 15.2 itself warns ("a mayor
  nivel, mayor consumo de IA y mayor necesidad de revisión humana").
- Building real Level 2+ analysis now would be a **materially larger and
  riskier undertaking** than what this roadmap step's own stated
  dependencies (INC-06 increment/bug/plan commands, INC-09/INC-10
  skills/technical-context indexes, INC-07 index rebuild — all
  mechanical, structural increments) suggest is in scope for Phase F.
  Nothing in the dependency chain requires deep semantic codebase
  understanding to exist yet. Scoping this pass to Level 0/1 keeps the
  increment's risk and AI-judgment profile consistent with every other
  increment closed so far in this roadmap (schema/mechanical work
  audited and extended, not open-ended AI-authored analysis).
- This is consistent with, not a deviation from, source doc 15.2 itself:
  the source doc defines levels as an explicit user-facing choice
  precisely so that a minimal level can be built and shipped before
  higher, costlier levels — it does not mandate building all five at
  once.

### Collapsed-role recommendation

**Recommendation: collapse the roadmap's stated 4-role sequence
(`bootstrap-analyzer -> docs-skills-writer -> index-engineer ->
validator-reviewer`) to 2 roles for a Level 0/1 pass: one
`bootstrap-implementer` (analysis + draft + write + index, in one
pass) followed by `validator-reviewer` (final review only, unchanged).**

The roadmap's originally-cited 4-role split (and even more so, source doc
15.3's literal 7-subagent chain) is designed for a task with genuinely
distinct concerns needing separate judgment: e.g. a `backend-analyzer`
producing different, non-mechanical insight than a `frontend-analyzer`,
handed off to a `context-writer` making independent drafting judgment
calls, then an `index-builder` making independent curation judgment
calls about what counts as `mandatory` vs `available`. That separation
earns its cost when each step involves real, distinct AI judgment.

At Level 0/1 scope specifically, this does not hold:

- **Analysis** (detect stack, enumerate repo structure, extract commands)
  and **drafting** (write a templated summary document from that
  detected data) are two steps of the *same* mechanical transformation,
  not two independent judgment calls — the draft content **is** the
  analysis output, reformatted as prose/Markdown. Splitting them into two
  handoff-based roles would add coordination overhead (an intermediate
  analysis-result artifact to serialize, pass, and re-parse) without a
  corresponding quality benefit, since neither step requires judgment the
  other role lacks.
- **Index-building** at this level is equally mechanical: for each
  drafted document, write one `available` entry
  (`{id, path, summary, tags}`) derived directly from the same detected
  data (e.g. `tags: ['backend']` if the doc came from a repo detected as
  backend-flavored). Per the "Fitness confirmation" section above and
  Q-bootstrap-1, this pass deliberately does **not** populate
  `mandatory.*` (a genuine curation judgment, correctly left to a human or
  a later promotion step) — so the one judgment call that would justify a
  separate `index-engineer` role (deciding what's mandatory) is exactly
  the part this pass does not do.
- A single `bootstrap-implementer` role doing analysis + draft + write +
  index in one pass is analogous to how this roadmap's own closed
  increments already treat comparably mechanical multi-step work as one
  role (e.g. INC-07's index-rebuild, INC-09's schema-writer +
  docs-skills-writer combination in practice) rather than insisting on
  maximal role separation regardless of actual task shape.
- `validator-reviewer` remains a distinct, separate role, unchanged from
  the roadmap's sequence: reviewing bootstrap output against acceptance
  criteria (correct draft/review marking, valid `TechnicalContextIndex`
  shape, referential consistency between index paths and actually-written
  files, no `mandatory.*` entries populated) is a genuinely independent
  check that benefits from being a separate pass, consistent with the
  parent-orchestrator's own "fresh review rule" for adversarial/
  independent review after implementation.
- This collapse is explicitly scoped to Level 0/1. If a future increment
  builds Level 2+ (genuine architectural/semantic analysis), re-splitting
  into more roles (e.g. a real `architecture-summarizer` doing judgment
  work distinct from a mechanical `context-writer`) would likely be
  justified again — this recommendation is not a permanent collapse of
  the roadmap's role vocabulary, only a right-sizing for this specific
  pass's actual complexity.

### CLI surface design

**Recommendation: `axiom bootstrap from-code --level minimal|basic`**,
following the established `axiom <namespace> <verb>` pattern confirmed by
reading `Axiom/apps/cli/src/commands/skills.ts` (`axiom skills list`,
`... refresh`, `... drift`, `... apply`) and
`Axiom/apps/cli/src/commands/index-cmd.ts` (`axiom index rebuild`,
`... validate`).

- **Namespace**: `bootstrap` (new, additive namespace — matches source
  doc 15's own name for the concept and the roadmap's own command name,
  `bootstrap-from-code`, split into namespace + verb per the existing
  convention rather than kept as one hyphenated token).
- **Verb**: `from-code`, mirroring source doc 15.3's own literal name for
  Via A and leaving room for a sibling `axiom bootstrap from-legacy-sdd`
  (Via B, INC-19) under the same namespace later — consistent with how
  `axiom skills`/`axiom index` group related verbs under one namespace
  rather than inventing a new top-level command per verb.
- **`--level minimal|basic`**: closed-enum flag scoped to exactly the two
  levels this pass builds (mirrors this audit's own recommendation not to
  build Level 2+ yet — the flag should not expose `standard|deep|
  exhaustive` values that do nothing). Default: `minimal` (cheapest,
  safest default, consistent with the source doc's own warning to not
  default users into higher AI-consumption/review-burden levels).
- **Target repo/scope resolution**: should reuse the existing
  `specScope` resolution pattern (per `general-spec.md`'s
  "Technical-context index" section and INC-10's audit) rather than
  inventing a new one — the command needs to resolve both "which repo(s)
  am I analyzing" (likely the current project's registered
  `roleCodeRepositories`/`target` repo per `@axiom/topology`) and "where
  do I write the technical-context output" (the resolved `specScope`).
  Confirming the exact resolution call is the next role's (implementer's)
  task, not decided here.
- **Output contract**: on success, the command should report which files
  it wrote (documents + index) and explicitly print a reminder that
  output is in draft/review state and requires human approval — mirroring
  addendum section 15's requirement and the existing convention of
  commands like `axiom context refresh` reporting `generatedFiles` counts
  and status lines.
- Not designed here (left to the next role): exact flag list beyond
  `--level`, e.g. whether a `--role`/`--repo-kind` filter is needed for
  multi-repo/multi-role projects, and whether `--dry-run` should exist
  (recommended, consistent with `axiom upgrade --dry-run`'s precedent,
  but not decided as in-scope for this audit).
